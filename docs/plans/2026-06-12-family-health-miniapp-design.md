# 家庭健康档案与体检报告解读小程序设计文档

## 1. 产品定位

本产品是一个面向家庭场景的微信小程序，用于帮助用户看懂体检报告和化验单，管理家人健康档案，持续追踪重点健康指标，并提供复查、复诊与生活方式提醒。

一句话定位：

> 帮全家人看懂体检报告，持续追踪重点健康指标。

产品不定位为 AI 医生、在线诊疗工具、疾病诊断工具、处方建议工具或治疗方案生成器。

---

## 2. 目标用户

### 2.1 核心用户

- 关注自己和家人健康的普通家庭用户
- 需要帮父母、伴侣、孩子管理体检报告和复查提醒的人
- 看不懂体检报告、化验单指标，希望获得通俗解释的人

### 2.2 当前开发约束

- 产品第一阶段仅面向自家使用
- 单微信账户管理所有家庭成员数据
- 不做多人家庭共享权限
- 不做医生端、社区、药品管理等重功能

---

## 3. 核心使用场景

### 3.1 看懂报告

用户上传体检报告或化验单，小程序帮助用户解释指标含义、异常项、可能相关因素、生活方式建议和复查建议。

### 3.2 管理家人健康档案

用户可以为自己、父母、伴侣、孩子创建健康档案，记录基础信息、报告、化验单、关键指标和历史提醒。

### 3.3 追踪重点指标趋势

系统持续记录血压、血糖、血脂、尿酸、肝肾功能等重点指标，并以趋势图展示变化。

### 3.4 复查和生活提醒

系统根据用户手动设置或 AI 解读建议，提醒用户复查、复诊，或关注饮食、运动、作息等生活习惯。

## 4. 信息架构

底部 Tab 保留 3 个：

```text
首页 / 家人 / 更多
```

### 4.1 首页

首页是家庭健康总览，不是功能入口堆叠页。

主要内容：

- 今日重点
- 异常指标提醒
- 复查 / 复诊提醒
- 饮食与生活建议
- 最近报告入口
- 快捷操作

### 4.2 家人

家人页是家庭成员健康档案索引页，以已有档案展示为主，以新增成员为辅。

主要内容：

- 默认「我」的健康档案
- 已创建家人的健康档案卡片
- 添加家人入口
- 成员详情页入口
- 报告上传入口
- 指标趋势入口

### 4.3 更多

更多页承载设置与低频功能。

主要内容：

- 数据导出
- 提醒管理
- AI 设置
- 隐私说明
- 免责声明
- 关于产品

## 5. 首页设计

首页只回答 3 个问题：

1. 谁需要关注？
2. 哪个指标在变化？
3. 下一步该做什么？

### 5.1 首页信息优先级

```text
1. 今日最重要的风险
2. 即将到期的提醒
3. 近期异常指标趋势
4. 最近一次报告入口
5. 快捷操作
```

### 5.2 首屏结构

```text
┌──────────────────────┐
│ 家庭健康总览          │
│ 今日重点：父亲血压连续偏高 │
│ 建议：今晚继续监测，避免高盐 │
├──────────────────────┤
│ 待办提醒              │
│ 06/15 母亲复查血糖     │
│ 06/20 父亲心内科复诊   │
├──────────────────────┤
│ 重点指标趋势          │
│ 血压 ↑  血糖 →  尿酸 ↑ │
├──────────────────────┤
│ 最近报告              │
│ 2026-06-01 父亲体检报告 │
│ 点击查看解读          │
├──────────────────────┤
│ 快捷操作              │
│ + 录入报告  + 新增提醒  │
└──────────────────────┘
```

### 5.3 首页设计原则

- 风险优先：异常指标和复诊提醒排在前面
- 减少认知负担：首页只展示下一步行动
- 颜色克制：异常使用柔和橙红，不使用大面积警报红
- 卡片化：每个模块独立，方便后续动效和渐进展开

## 6. 家人页设计

家人页不是“添加家人页”，而是“健康档案入口页”。添加家人只在无数据或用户主动需要时出现。

### 6.1 首次使用状态

首次进入时，系统默认存在一个「我」的身份，但还没有健康档案。

```text
家人

┌────────────────────────┐
│  先建立你的健康档案       │
│  上传体检报告或录入基础信息 │
│                        │
│  [上传我的体检报告]       │
│  [完善我的基础信息]       │
└────────────────────────┘

┌────────────────────────┐
│  也可以管理家人健康       │
│  为父母、伴侣、孩子建立档案 │
│                        │
│  [添加家人]              │
└────────────────────────┘
```

主按钮应是「上传我的体检报告」，因为产品核心是体检报告解读，而不是通讯录式成员管理。

### 6.2 已有自己档案但没有其他家人

```text
家人

┌────────────────────────┐
│  我                     │
│  最近报告：2026-06-01    │
│  重点关注：尿酸、血脂      │
│  [查看档案]              │
└────────────────────────┘

+ 添加父母 / 伴侣 / 孩子
```

此时添加家人入口应弱化，不抢主视觉。

### 6.3 已有多个家人

```text
家人

┌────────────────────────┐
│  我                     │
│  尿酸偏高｜待复查         │
└────────────────────────┘

┌────────────────────────┐
│  父亲                   │
│  血压关注中｜3 天后复诊    │
└────────────────────────┘

┌────────────────────────┐
│  母亲                   │
│  血糖稳定｜暂无提醒        │
└────────────────────────┘

右上角：+
```

### 6.4 家人页核心原则

```text
无档案时：引导创建「我」的健康档案
有档案时：优先展示已有家人
添加家人：始终存在，但不打断主流程
```

## 7. 成员详情页设计

成员详情页用于集中展示某个家庭成员的健康状态、报告、指标趋势和提醒。

### 7.1 页面结构

```text
1. 个人概览
   - 基础信息
   - 既往病史
   - 风险标签

2. 最近报告
   - 报告列表
   - 报告类型
   - 查看原图 / PDF
   - AI 解读入口

3. 重点指标趋势
   - 折线图
   - 正常范围阴影
   - 异常点标注

4. 建议与提醒
   - 复查时间
   - 饮食提醒
   - 运动 / 作息建议
```

### 7.2 交互重点

- 进入成员详情后，默认展示最近一次报告和当前风险
- AI 解读放在报告详情页中，不单独作为一级入口
- 同一成员下所有记录按时间线展示，方便追踪趋势

## 8. 报告上传与解读流程

报告上传是产品核心链路。设计原则是：先上传，后确认，再沉淀为档案。

### 8.1 推荐流程

```text
选择成员
→ 上传体检报告 / 化验单
→ 系统 OCR / 结构化识别
→ AI 生成报告解读
→ 用户确认关键指标
→ 保存为该成员档案
→ 自动进入趋势统计与提醒
```

### 8.2 MVP 阶段处理方式

MVP 阶段不追求完整 OCR 自动解析，优先支持：

- 图片 / PDF 上传
- 原始文件保存
- 手动录入关键指标
- AI 基于用户输入或报告文本进行解读
- 用户确认后再写入结构化指标库

### 8.3 AI 解读页固定结构

```text
1. 报告摘要
2. 异常指标
3. 重点关注
4. 可能影响因素
5. 生活方式建议
6. 复查 / 就医提醒
7. 免责声明
```

### 8.4 指标确认页

AI 或 OCR 识别后的指标必须由用户确认：

- 指标名是否正确
- 数值是否正确
- 单位是否正确
- 参考范围是否正确

确认后才写入数据库，作为趋势图和提醒的数据源。

## 9. 数据模型设计

核心数据对象拆成 5 类：

```text
1. 家庭成员 member
2. 体检 / 化验报告 report
3. 指标明细 metric_record
4. 提醒 reminder
5. AI 解读记录 interpretation
```

### 9.1 成员 member

用于存储基础档案。

字段建议：

- 姓名 / 昵称
- 关系
- 性别
- 生日
- 身高
- 体重
- BMI
- 既往病史
- 过敏史
- 常用就诊医院
- 关注重点标签：血压 / 血糖 / 血脂 / 尿酸等

### 9.2 报告 report

一份体检单或化验单对应一条报告记录。

字段建议：

- 成员 ID
- 报告类型：体检 / 门诊 / 化验 / 影像
- 医院
- 科室
- 报告日期
- 原始文件地址
- 报告摘要
- AI 解读结果 ID

### 9.3 指标明细 metric_record

每个异常或关键指标单独记录，便于趋势统计。

字段建议：

- 成员 ID
- 报告 ID
- 指标名
- 指标编码
- 数值
- 单位
- 参考范围
- 是否异常
- 指标分类：血常规 / 肝功 / 肾功 / 血脂 / 代谢等
- 检测日期

### 9.4 提醒 reminder

用于复查、复诊和日常习惯提醒。

字段建议：

- 成员 ID
- 关联报告 ID
- 关联指标 ID
- 提醒类型：复诊 / 复查 / 生活习惯
- 提醒标题
- 提醒时间
- 重复规则：一次性 / 每天 / 每周
- 备注
- 状态：待完成 / 已完成 / 已取消

### 9.5 AI 解读 interpretation

保存每次 AI 输出，便于追溯。

字段建议：

- 成员 ID
- 报告 ID
- 输入内容摘要
- 提问内容
- 输出建议
- 风险等级
- 创建时间
- 是否已查看

## 10. 提醒系统设计

第一版只做 3 类提醒：

```text
1. 复查提醒
2. 复诊提醒
3. 日常习惯提醒
```

### 10.1 复查提醒

来源：

- AI 解读报告后建议生成
- 用户手动创建

示例：

```text
尿酸偏高
建议 1-3 个月后复查
→ 创建复查提醒：2026-09-01
```

### 10.2 复诊提醒

适合门诊病历、医生建议、慢病管理。

示例：

```text
父亲：心内科复诊
时间：2026-06-20
地点：瑞金医院
备注：带上近期血压记录
```

### 10.3 日常习惯提醒

只保留高频、明确、可执行事项。

示例：

- 晚上 8 点记录血压
- 每天饮水
- 每周运动 3 次
- 低盐饮食提醒
- 控制高嘌呤食物

MVP 阶段只支持：

```text
一次性提醒
每天重复
每周重复
```

### 10.4 首页提醒排序

```text
逾期 > 今日 > 未来 7 天 > 普通习惯
```

### 10.5 通知方式

微信小程序不能像 App 一样随意后台推送。第一版使用：

```text
小程序内提醒 + 订阅消息
```

订阅消息需要用户主动授权。未授权时，提醒仍应在首页和更多页展示。

## 11. AI 解读与问答边界

AI 功能定位为：

> 帮用户看懂报告，而不是替用户诊断疾病。

第一版只做两个受控入口：

```text
1. 报告解读：围绕某一份体检报告 / 化验单
2. 档案问答：围绕某个家人的历史健康档案
```

### 11.1 AI 可以做

```text
解释指标含义
指出异常指标
总结趋势变化
提示可能相关生活因素
建议复查或咨询医生
生成饮食、运动、作息建议
```

### 11.2 AI 不应该做

```text
确诊疾病
推荐处方药
调整药物剂量
否定医生建议
承诺治疗效果
替代线下检查
```

### 11.3 固定免责声明

所有 AI 回答末尾固定附带：

> 内容仅用于帮助理解体检报告和管理健康档案，不构成诊断或治疗建议。如指标持续异常或出现不适，请咨询医生。

## 12. 前端技术方案

### 12.1 推荐路线

```text
微信小程序原生框架
+ TypeScript
+ WXSS / SCSS
+ 自定义健康类 UI 设计系统
+ TDesign / Vant 少量基础组件
+ ECharts for Weixin
+ Lottie 小程序版
```

### 12.2 选型理由

- 原生小程序性能最好，微信能力支持最完整
- TypeScript 提高复杂数据结构的可靠性
- 自定义 UI 保留设计感和品牌调性
- TDesign / Vant 只用于表单、弹窗、Toast、日期选择等基础能力
- ECharts 用于指标趋势图
- Lottie 用于 AI 解读中、空状态、完成反馈等轻量动效

### 12.3 视觉方向

关键词：

```text
温和、可信、清爽、低压力、长期陪伴
```

建议：

- 主色：柔和蓝绿 / 青绿色
- 辅色：浅米白、淡蓝灰
- 警示色：柔和橙红
- 背景：低饱和暖白
- 卡片：圆角、大留白、轻阴影

## 13. 技术架构

```text
┌─────────────────────────────────────────┐
│            微信小程序前端                  │
│  原生框架 + TypeScript + 自定义 UI         │
│  ECharts + Lottie                        │
└──────────────┬──────────────────────────┘
               │ wx.cloud API
┌──────────────▼──────────────────────────┐
│         微信云开发 CloudBase              │
│  ├─ 云数据库：成员、报告、指标、提醒、解读    │
│  ├─ 云存储：体检报告 PDF / 图片             │
│  ├─ 云函数：业务逻辑 + 调 AI API           │
│  └─ 身份认证：微信登录                     │
└──────────────┬──────────────────────────┘
               │ HTTPS
┌──────────────▼──────────────────────────┐
│         外部 AI API                       │
│  DeepSeek / 通义千问 / GLM                │
│  → 接收结构化健康数据 + 报告文本            │
│  → 返回结构化解读与建议                    │
└─────────────────────────────────────────┘
```

### 13.1 后端选择

第一版使用微信云开发，不自建服务器。

原因：

- 免运维
- 不需要单独部署服务器
- 天然适配微信登录、小程序数据库和云存储
- 适合个人开发者快速验证

### 13.2 AI 接入方式

云函数中调用外部 AI API。

推荐优先级：

```text
1. DeepSeek：性价比高，API 兼容性好
2. 通义千问：国内生态稳定
3. GLM：备选
```

实际选择需在开发前根据 2026 年最新价格、稳定性和接口政策重新确认。

## 14. MVP 分期

### 14.1 MVP 目标

让用户完成主路径：

```text
创建自己的档案
→ 上传一份体检报告 / 化验单
→ AI 帮忙解读
→ 用户确认重点指标
→ 保存到健康档案
→ 首页看到异常指标和提醒
```

### 14.2 MVP 必做功能

```text
1. 微信登录
2. 默认创建「我」这个成员
3. 添加家人
4. 上传图片 / PDF 报告
5. 手动录入关键指标
6. AI 解读报告
7. 保存报告记录
8. 查看成员健康档案
9. 创建复查 / 复诊提醒
10. 首页展示今日重点与待办提醒
```

### 14.3 MVP 暂不做

```text
1. 自动 OCR 完整识别
2. 医院报告格式自动解析
3. 多账户家庭共享
4. 医生端
5. 社区内容
6. 药品管理
7. 复杂健康评分
8. 复杂订阅通知系统
```

### 14.4 Phase 1：健康档案基础版

目标：不用 AI，也能管理档案。

```text
- 创建「我」
- 添加家人
- 录入基础信息
- 上传报告文件
- 手动录入指标
- 查看报告列表
- 查看成员详情
```

### 14.5 Phase 2：AI 报告解读版

目标：让产品真正有价值。

```text
- 云函数接入 AI API
- 报告解读 Prompt 模板
- 输出结构化解读
- 异常指标解释
- 生活方式建议
- 复查建议
- 保存 AI 解读记录
```

### 14.6 Phase 3：趋势与提醒版

目标：让用户持续回来。

```text
- 指标趋势图
- 异常指标聚合
- 首页今日重点
- 复查 / 复诊提醒
- 小程序订阅消息
- 更多页提醒管理
```

### 14.7 推荐开发顺序

```text
1. 初始化小程序 + 云开发
2. 搭建 3 Tab：首页 / 家人 / 更多
3. 建立数据库集合
4. 做「我」的默认档案
5. 做添加家人
6. 做报告上传
7. 做手动指标录入
8. 做成员详情页
9. 接入 AI 解读
10. 做提醒系统
11. 做趋势图
12. 做视觉和动效打磨
```

## 15. 风险与合规边界

### 15.1 产品边界

产品只做：

```text
体检报告解读
健康档案管理
重点指标趋势追踪
复查提醒
生活方式建议
```

产品不做：

```text
AI 医生
在线诊疗
疾病诊断
处方建议
治疗方案生成
```

### 15.2 数据安全边界

健康数据属于敏感信息。第一版至少做到：

```text
1. 明确告知用户数据用途
2. 原始报告文件存储在微信云存储
3. AI 解读前提示报告内容会发送给第三方模型服务
4. 允许用户删除报告、指标、成员档案
5. 不做公开分享
6. 不做社区讨论
7. 不默认上传通讯录或家庭成员隐私信息
```

AI 调用前建议显性提示：

> 报告解读需要调用第三方 AI 服务，报告文本可能会被发送至模型接口用于生成解读。

### 15.3 技术风险与处理方式

| 风险 | 处理方式 |
|---|---|
| OCR 识别错误 | MVP 先手动录入关键指标；OCR 结果必须用户确认 |
| AI 胡说 | 固定结构化 Prompt；限制诊断和用药；保留免责声明 |
| 医院报告格式差异大 | 不追求全自动解析；先做关键指标录入 |
| 提醒不可靠 | 小程序内提醒为主，订阅消息为辅 |
| 数据泄露 | 不做多账户共享；不做公开链接；删除能力优先实现 |
| 功能过重 | 严格按 Phase 1 → 2 → 3 推进 |
