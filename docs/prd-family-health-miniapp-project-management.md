# IeTecho 家庭健康小程序 PRD

## AI 速读卡

- 产品一句话：IeTecho 是一款家庭体检报告解读与健康档案微信小程序，让用户看懂报告、确认指标、追踪趋势和管理复查提醒，而无需自己整理零散体检单。
- 核心循环：上传报告 → AI 解读 → 用户确认指标 → 保存档案 → 首页提醒下一步 → 下次复查继续追踪。
- 目标平台：微信小程序，P0 先做静态 Mock + 原生小程序结构，P1 接微信云开发。
- 硬约束：不做 AI 医生；不诊断、不开药、不生成治疗方案；未确认指标不得进入趋势统计。
- 推荐默认：单账户管理全家数据；原生小程序 + TypeScript + WXSS token + 微信云开发。
- 发挥空间：家人档案卡视觉、指标批注动效、状态环展开、空态文案、报告解读排版。
- P0 验收：微信开发者工具可打开；首页 / 家人 / 报告 / 我的可用；报告上传到指标确认的静态路径走通。
- 最容易翻车：把产品做成 AI 问诊；把健康状态环做成伪健康分；跳过用户确认直接入库。
- 超预期机会：报告批注卡逐条点亮、家人档案卡浮入、复查提醒自动草稿、年度健康摘要。

## 第一章：产品概述

IeTecho 是一款家庭健康档案与体检报告解读微信小程序，让关注家人健康的普通用户能够上传报告、看懂指标、确认关键数据并持续追踪变化，而无需手动整理零散体检报告和复查时间。

### 1.1 差异化对比表

> 竞品事实未联网核验；以下基于常见健康 App、体检机构报告系统、医院互联网平台和个人笔记工具的结构性差异推断。

| 功能 | 竞品 | 本产品 | 实现方式 |
|---|---|---|---|
| 家庭成员维度 | 多数健康 App 以个人账户为中心，家人管理通常弱化 | 单账户管理「我、父亲、母亲」等家人档案 | `member` 作为一等对象，报告、指标、提醒均挂到成员下 |
| 报告解读闭环 | 体检机构给单次报告解释，长期追踪通常断裂 | 上传报告后解释、确认指标、进入趋势和提醒 | 报告 `report` 与指标 `metric_record` 分离，确认后才入库 |
| AI 输出边界 | 问诊类产品容易把用户引向诊疗问答 | AI 只帮用户看懂报告，不替代医生 | 固定解读结构 + 免责声明 + 禁止诊断/处方 Prompt |
| 趋势可靠性 | OCR 或 AI 识别后若直接入库，容易污染趋势 | 用户确认关键指标后才进入趋势统计 | 指标状态包含 `pending / confirmed / ignored / modified` |
| 提醒场景 | 日历或备忘录只提醒时间，不懂指标上下文 | 复查、复诊、生活提醒和报告/指标关联 | `reminder` 可关联成员、报告、指标 |

### 1.2 三类用户画像

#### 用户画像 A：给父母管理体检的人

- 角色：中国大陆城市家庭中的成年子女。
- 核心目标：帮父母看懂体检报告，知道哪些指标要复查。
- 对现有工具最大的不满：体检报告、微信聊天截图、医院记录分散在不同地方。
- 让他们愿意切换的那一个功能：父母档案卡显示「谁需要关注、何时复查、带什么记录」。

#### 用户画像 B：自己定期体检但看不懂指标的人

- 角色：有年度体检习惯的普通用户。
- 核心目标：知道尿酸、血脂、血糖等指标偏高是什么意思，以及下次该关注什么。
- 对现有工具最大的不满：报告里异常项很多，但解释不够通俗，也缺少连续趋势。
- 让他们愿意切换的那一个功能：报告批注卡用通俗语言解释指标，并保存为长期趋势。

#### 用户画像 C：慢病初期管理家庭成员的人

- 角色：家里有人血压、血糖、血脂需要长期关注。
- 核心目标：记录复查、复诊和生活提醒，避免忘记随访。
- 对现有工具最大的不满：日历提醒没有健康上下文，医生建议和指标变化断开。
- 让他们愿意切换的那一个功能：提醒卡自动关联成员、指标和报告。

### 1.3 可行性边界

| 在范围内（及原因） | 明确排除在外（及原因） |
|---|---|
| 微信小程序静态 Mock：验证页面结构与视觉方向 | P0 不接真实云开发：避免过早陷入账号、环境、权限问题 |
| 单账户家庭档案：符合自用和 MVP 验证 | P0 不做多人共享：权限、隐私和家庭关系模型会显著复杂化 |
| 报告解读辅助：帮助理解指标含义和建议复查 | AI 医生、诊断、处方、治疗方案：存在合规和责任风险 |
| 手动确认指标：保证趋势数据可靠 | OCR 自动全量解析：医院报告格式差异大，P0 容易失控 |
| 小程序内提醒和订阅消息预留 | 承诺后台强推：微信订阅消息需要用户授权，不可无感推送 |
| 数据导出作为后续能力 | P0 不做 PDF/Excel 导出：不是验证主循环的必要条件 |

### 1.4 约束分层

| 硬约束 | 推荐默认 | 发挥空间 |
|---|---|---|
| 不使用「AI 医生」「诊断中」「治疗方案」等定位或文案 | 原生微信小程序 + TypeScript | 卡片排版、空态文案、动效节奏可继续打磨 |
| AI 结果必须有免责声明 | 单微信账户管理所有成员 | 家人头像可用关系色块、图标或后续真实头像 |
| 未确认指标不能进入趋势统计 | P0 用 mock 数据，P1 接微信云开发 | 报告解读卡的视觉表达可更有批注感 |
| 健康状态环不能展示健康分或疾病概率 | 4 Tab：首页 / 家人 / 报告 / 我的 | 状态环可做轻交互，但只展示事实聚合 |
| 上传页必须提示第三方 AI 服务会处理报告内容 | 先手动录入/确认关键指标，再考虑 OCR | 指标知识库可后续扩展为可维护内容库 |


### 2.2 移动端主界面布局

```text
+----------------------------------+
| 家庭健康总览                     |  约 10%
| 今天先看最需要关注的人和下一步行动 |
+----------------------------------+
| 今日重点卡                       |  约 24%
| 父亲｜血压连续 3 次偏高           |
| 建议今晚继续记录血压              |
| 「查看父亲档案」                   |
+----------------------------------+
| 待办提醒                         |  约 18%
| 06/15 父亲心内科复诊              |
| 09/01 我复查尿酸                  |
+----------------------------------+
| 重点指标趋势摘要                  |  约 18%
| 尿酸 ↑  血压 ↑  血糖 →            |
+----------------------------------+
| 最近报告 + 快捷操作               |  约 20%
+----------------------------------+
| 首页        家人        更多       |  固定 Tab
+----------------------------------+
```

布局理由：用户打开健康工具时首先要知道「谁需要关注」和「下一步做什么」，不是浏览功能列表。因此首页采用状态优先的仪表盘结构。

### 2.3 核心流程

```text
用户点击「上传报告」
    |
    v
选择成员 / 默认「我」
    |
    v
报告上传占位 + AI 调用提示
    |
    +-- 用户点击「先保存，不解读」 --> 成员详情显示未解读报告
    |
    +-- 用户点击「开始解读」
            |
            v
        解读中页
            |
            +-- 解读失败 --> 保留原始报告，可重新解读或手动录入
            |
            +-- 解读完成 --> 报告解读结果
                                |
                                v
                            指标确认
                                |
                                +-- 保存失败 --> 保留已编辑指标，允许重试
                                |
                                +-- 保存成功 --> 成员详情 / 首页刷新今日重点
```

失败路径 1：仪表盘数据读取失败 → 显示错误卡「暂时无法加载家庭健康总览」→ 用户点重试。

#### c) 状态清单

| 名称 | 触发条件 | 视觉标识 | 退出条件 |
|---|---|---|---|
| 首次使用 | 无完整档案 | 暖白引导卡 + 主按钮 | 上传报告或完善基础信息 |
| 今日重点 | 有今日/逾期提醒或异常指标 | 顶部重点卡 | 用户进入详情或标记完成 |
| 近期平稳 | 无异常且无到期提醒 | 低压力说明 | 新增报告或提醒 |
| 加载失败 | 数据读取失败 | 错误卡 | 重试成功 |

#### d) 依赖关系

读取：`Dashboard`、`Member`、`Reminder`、`Report`、趋势摘要。  
写入：P0 不写；P1 可触发提醒完成状态。  
数据流：云数据库 / mock 数据 → `getDashboard` 聚合 → 首页渲染。

## 第四章：超越竞品的差异化功能

### 第 4.1 节 家人档案卡作为主入口

1. 竞品为何没有这个功能：基于公开信息的推断，体检机构和医院系统通常以单次报告或单个患者就诊记录为中心，不以「家庭健康负责人」为主使用场景。
2. 本产品如何实现：把 `member` 作为所有报告、指标、提醒的聚合根；首页和家人页都围绕成员状态组织。
3. 交互流程：

```text
打开家人页
  |
  v
看到父亲 / 我 / 母亲档案卡
  |
  +-- 父亲卡显示「3 天后复诊」 --> 点击进入父亲详情
  |
  +-- 我卡显示「尿酸待复查」 --> 点击进入我的趋势
```

4. 风险与应对：卡片信息过载 → 每张卡只展示最近报告、关注标签、下一步行动。

### 第 4.2 节 指标确认后才进入趋势

1. 竞品为何没有这个功能：自动化报告解读产品倾向强调「上传即得到结果」，但健康趋势对数据准确性更敏感。
2. 本产品如何实现：`metric_record.confirmationState` 记录 `pending / confirmed / modified / ignored`，只有 confirmed 或 modified 进入趋势统计。
3. 交互流程：

```text
AI 识别尿酸 468
  |
  v
用户看到「待确认」
  |
  +-- 确认 --> 写入趋势
  +-- 编辑 --> 标记已修改后写入趋势
  +-- 忽略 --> 不写入趋势
```

4. 风险与应对：用户觉得确认麻烦 → 默认只要求确认异常指标和重点指标，不要求确认所有正常项。

### 第 4.3 节 报告批注式解读

1. 竞品为何没有这个功能：传统体检报告解释多以长文本说明为主，用户难以定位到具体异常项。
2. 本产品如何实现：用 `MetricAnnotationCard` 承载指标名、数值、参考范围、通俗解释、下一步建议。
3. 交互流程：

```text
报告解读结果
  |
  v
异常指标卡逐条显示
  |
  +-- 用户点尿酸卡 --> 展开解释和建议
  +-- 用户点确认 --> 进入指标确认页
```

4. 风险与应对：批注像诊断 → 文案只解释指标含义、可能因素和复查建议，不给疾病结论。

## 第十一章：开发者交接说明

### a) 实现顺序建议

你先构建静态 Mock，不要先接云开发。原因是当前最大不确定性不是后端，而是页面结构、健康产品边界和视觉质感。静态 Mock 走通后，再接 P1 的云数据库、云存储和 AI 云函数。

建议顺序：

1. 保持当前 `miniprogram/` 原生结构。
2. 先修复微信开发者工具编译问题。
3. 打磨首页、家人页、成员详情的视觉层级。
4. 走通上传报告 → 解读中 → 解读结果 → 指标确认 → 成员详情。
5. 再接云开发数据持久化。
6. 最后接 AI 解读和趋势图。

### b) 最可能导致返工的三个决策

#### 决策 1：是否把 AI 问答做成一级入口

- 安全默认：不做一级入口，AI 嵌入报告解读和成员档案。
- 改变信号：如果用户真实使用中主要从自由问答进入，而不是从报告上传进入，再考虑增加问答入口。

#### 决策 2：指标是否自动入库

- 安全默认：不自动入库，必须用户确认。
- 改变信号：OCR/AI 准确率在同类报告上经过验证达到 99% 以上，且用户明确愿意自动保存。

#### 决策 3：技术栈是否改 Taro / uni-app

- 安全默认：原生微信小程序。
- 改变信号：明确需要 H5、App 或多个小程序平台同步上线。

### c) 哪里要严格，哪里可以灵活

| 章节 | 约束 / 建议 | 原因 |
